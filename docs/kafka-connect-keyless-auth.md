# Kafka Connect keyless GCP auth

Kafka Connect should not rely on a long-lived service account key generated by
Terraform. The BigQuery sink connector is configured with
`keySource=APPLICATION_DEFAULT`, so the Connect worker resolves credentials from
the runtime environment.

## Local development

1. Create local Application Default Credentials:

   ```bash
   gcloud auth application-default login
   ```

2. Start Kafka Connect with the ADC file mounted read-only. The local streaming
   compose file expects `GOOGLE_APPLICATION_CREDENTIALS` to point at that local
   ADC file and mounts it as `/gcp/adc.json` inside the container.

3. Do not commit the ADC file, service account keys, or anything under
   `local_scripts/streaming/secrets/`.

## Production

Production Kafka Connect should run as a Kubernetes workload with Workload
Identity:

1. Run the Connect pod with a Kubernetes ServiceAccount.
2. Bind that KSA to the Google Service Account used for streaming.
3. Grant that GSA the BigQuery/GCS roles needed by the sink connector.
4. Do not set `GOOGLE_APPLICATION_CREDENTIALS` to a key JSON file.
5. Verify the pod resolves the expected GSA from the metadata server before
   deleting or revoking old local keys.

Terraform no longer creates `google_service_account_key` for Kafka Connect and
does not write `kafka_connect_key.json` into the streaming workspace.
